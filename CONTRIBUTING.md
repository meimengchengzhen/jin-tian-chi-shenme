# 贡献指南

感谢你愿意为「今天吃什么」加菜 🍳。本项目是一个**纯前端、无登录、无后端**的家常菜随机器原型，定位是「打开就能用、好分享、好 fork」的 MVP。所有贡献都欢迎，特别是菜谱数据、UI 细节打磨和算法迭代。

阅读顺序建议：

1. 看一遍 [README.md](./README.md) 了解整体结构。
2. 跑一遍本地开发环境（`npm install && npm run dev`）。
3. 找一类自己想做的贡献，按下面对应小节走。

---

## 一、贡献菜谱（最常见）

菜谱是这个项目最欢迎的贡献类型。所有内置菜谱都集中在 **一个文件** 里：

```
client/src/data/recipes.ts
```

### 步骤

1. **Fork** 本仓库并基于 `master` 切分支：`recipe/<菜名拼音>`，例如 `recipe/yu-xiang-rou-si`。
2. 在 `RECIPES` 数组末尾追加一个新条目（按 `course` 分类靠近放）。
3. 字段务必符合 `recipes.schema.json` —— 可以用任意 JSON Schema 校验工具（`ajv`、VSCode 内置等）跑一遍。
4. 跑 `npm run check`（TypeScript 检查）和 `npm run build`，确保没破坏构建。
5. 本地 `npm run dev` 启动后多点几次「换一组」，确认你新加的菜会被抽到、卡片显示正常。
6. 提 PR，标题以 `recipe:` 开头，例如 `recipe: 加入鱼香肉丝`。

### 写菜谱的内容规范

- **`name`**：标准中文菜名，2-12 字，不要加 emoji、不要加「的做法」「家常版」之类后缀。
- **`reason`**：一句话推荐语，**≤ 40 字**，告诉用户为什么今天可以吃这个。避免「美味可口、营养丰富」这种万能套话。
- **`steps`**：3-6 步最佳。每一步是动作 + 关键信息，例如「热油滑炒鸡蛋至嫩」。不要写成菜谱书的长段落。
- **`ingredients`**：
  - `qty` 写**现实买菜会用到**的描述，例如 `300g`、`2 根`、`1 把`、`适量`，不要写「100 克 ± 5g」这种实验室口吻。
  - `category` 三选一，决定它出现在「今日买菜清单」的哪一栏。
  - 多种调料可以合并：`{ name: "生抽 / 老抽 / 料酒", qty: "各适量", category: "调味/主食" }`。
- **`contains`**：标记**这道菜含有**的限制因素。例如这道菜里有牛肉，就填 `"无牛肉"`，含花生填 `"无花生"`。`素食` 是反向标签：表示本菜符合素食可在素食模式下保留。
- **`tastes`**：选**最显著**的 1-3 个口味标签，太多反而让推荐打分失真。
- **`difficulty`** 与 **`timeMinutes`** 请实际估，别把红烧肉写成 15 分钟。

### 不收的菜（请提前避免）

- 网图食谱搬运、未亲手做过的「网红菜」。
- 含有版权争议的图片或商标（项目目前不存图片）。
- 涉及野味、违禁食材或地域歧视的命名。
- 步骤超过 12 条的复杂筵席菜 —— 本项目只做家常。

## 二、提交功能 / Bug

### Bug

提 issue，模板：

```
[Bug] 简要描述
- 复现步骤：
- 预期：
- 实际：
- 环境（浏览器 / 系统 / 屏幕宽度）：
- 截图（可选）：
```

### 功能建议

提 issue，模板：

```
[Feature] 简要描述
- 解决什么场景：
- 一句话用户故事：作为 ___，我希望 ___，这样我可以 ___。
- 是否影响数据结构 / 算法 / UI：
```

确认会做之后再开 PR，避免做完才发现方向不一致。

### Pull Request

- 大改动先开 issue 讨论方向，再写代码。
- 一个 PR 聚焦一件事；混合「加菜 + 改算法 + 改样式」的 PR 会被拆。
- 描述里写清「为什么 / 改了什么 / 怎么验证」。

## 三、分支命名

| 类型     | 前缀         | 示例                                |
| -------- | ------------ | ----------------------------------- |
| 加菜     | `recipe/`    | `recipe/yu-xiang-rou-si`            |
| 新功能   | `feat/`      | `feat/markdown-export`              |
| Bug 修复 | `fix/`       | `fix/locked-card-vanishes-on-mobile`|
| 文档     | `docs/`      | `docs/contributing-zh`              |
| 重构     | `refactor/`  | `refactor/extract-recommend`        |
| 样式     | `style/`     | `style/tighten-mobile-padding`      |

提交信息推荐使用类似前缀，例如 `feat: 加入 Markdown 导出` / `fix: 修复锁定卡片在移动端被换掉`。中英均可，能让别人一眼看懂即可。

## 四、PR 检查清单

提交 PR 前请逐项确认：

- [ ] 已基于最新 `master` rebase。
- [ ] `npm run check` 通过（TypeScript 类型检查无报错）。
- [ ] `npm run build` 通过（产物在 `dist/public`）。
- [ ] 新增 / 修改的菜谱条目通过 `recipes.schema.json` 校验。
- [ ] 本地 `npm run dev` 跑过，至少在桌面 1280px+ 与移动 ~390px 各点过一次「今天吃什么 / 换一组 / 复制清单」。
- [ ] 没有引入 `localStorage` / `sessionStorage` / `cookie` —— 项目运行环境禁止使用。
- [ ] 没有新增不必要的依赖；如有新增，PR 描述中说明用途与体积影响。
- [ ] 截图：如改了 UI，PR 里贴 before / after 图。
- [ ] 文案与注释使用简体中文（详见下节）。

## 五、中文内容规范

项目面向中文用户，文案是产品体验的重要部分。请遵循：

- **统一简体中文**。代码标识符（变量名、文件名、commit 类型前缀）保持英文，注释、UI 文案、文档使用简体中文。
- **半角标点的取舍**：UI 文案和正式文档以**全角中文标点**为主（。，「」），代码注释允许用半角；中英混排时，英文 / 数字两侧加**一个半角空格**，例如「番茄 2 个 (约 300g)」「Vite + React」。
- **不用 emoji 替代图标**：项目刻意避免 emoji 文案；表达分类用 lucide-react 图标或自绘 SVG。README / Issue / PR 里允许少量使用。
- **避免「AI 味」用语**：不要「赋能、抓手、生态闭环」；推荐语避免「美味可口、营养丰富、老少皆宜」这种万能空话。
- **禁用词**：脏话、地域歧视、性别贬抑、政治敏感词、宗教争议词。涉及到饮食习惯（清真、素食等）请尊重事实陈述。
- **数字与单位**：优先用中文常用单位（斤、把、个、勺）配合克数，例如「2 根 (约 200g)」。
